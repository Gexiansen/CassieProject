# CassieProject schema v5 数据模型

## 目标

schema v5 将记账维度从传统消费分类改为支出的可调整程度。正式应用只读写 v5，不解析 v4 或更早结构；旧数据通过不被正式页面加载的一次性转换工具生成 v5 完整备份。v4 本地键和备份文件不会被网页覆盖或删除。

以下能力继续保留：整数分金额、UUID、版本校验、存储失败反馈、导入预览、导入前保护、异常数据锁定、删除撤销和离线使用。

## 本地存储

为避免手机端每次操作都重写全部数据，本地状态按职责分开保存：

- `cassie_records_v5`：支出记录。
- `cassie_settings_v5`：家庭成员和默认成员。
- `cassie_plans_v5`：预算、专项、目标、月结和无支出日期。
- `cassie_backup_meta_v5`：最近一次完整备份时间和摘要。

每个主存储对象都必须包含 `schemaVersion: 5` 和 `updatedAt`。内存中可以组合为 `appState`，但保存时只写发生变化的部分。

旧键只用于检测是否需要显示升级引导，不解析、不覆盖、不自动删除。用户可以导入转换后的 v5 完整备份，也可以二次确认后保留旧键并开始空白 v5 账本。

## 通用约束

- 所有业务 ID 都是 1～80 位的字母、数字、下划线或连字符。
- 新建业务对象使用 `crypto.randomUUID()`。
- 金额使用大于 0 的安全整数分；选填预算可为 `null`。
- 日期使用有效的 `YYYY-MM-DD`，月份使用有效的 `YYYY-MM`。
- 时间使用 ISO 8601 字符串。
- 用户文本在保存前去除首尾空白并按字段长度截断。
- 导入数据必须整体校验；发现无效引用时拒绝恢复，不静默丢弃记录。

## 支出记录

```json
{
  "id": "UUID",
  "date": "2026-07-19",
  "amountCents": 1234,
  "spendingType": "flexible",
  "beneficiaryId": "family",
  "projectId": "",
  "note": "晚餐食材",
  "createdAt": "2026-07-19T10:00:00.000Z",
  "updatedAt": "2026-07-19T10:00:00.000Z"
}
```

约束：

- v5 只有支出记录，不保存收入类型或传统消费分类。
- `spendingType` 必须是 `fixed`、`flexible`、`discretionary` 或 `exceptional`。
- `beneficiaryId` 必须引用设置中存在的获益方；停用成员仍可被历史记录引用。
- `projectId` 为空或引用规划中存在的正式专项。
- `note` 最多 20 个字符。
- 新记录不能使用已停用的成员。
- 转换工具遇到无法映射的 v4 分类时必须报告并停止，不得猜测。

## 家庭获益方

```json
{
  "id": "family",
  "name": "共同",
  "kind": "shared",
  "active": true
}
```

```json
{
  "id": "UUID",
  "name": "丈夫",
  "kind": "member",
  "active": true
}
```

约束：

- 数组顺序就是快速记账页的显示顺序。
- `family` 是保留 ID，`kind` 固定为 `shared`，不能停用或删除。
- 其他成员使用 `kind: "member"`，支持新增、改名、排序、停用和恢复。
- 名称为 1～6 个字符，同一设置中不能重名。
- 最多同时启用 8 个获益方，包括“共同”。
- `defaultBeneficiaryId` 必须引用启用的获益方；无效时回退 `family`。
- 历史记录引用的停用成员继续保留在设置中，不做硬删除。

默认设置预置“共同、妻子、丈夫、儿子”，沿用稳定 ID `family`、`wife`、`husband`、`son`，用户可以调整名称、顺序和启用状态。

## 支出属性

四种属性是 schema 定义，不进入用户设置：

| ID | 界面名称 | 决策含义 | 是否进入普通月份基线 |
| --- | --- | --- | --- |
| `fixed` | 固定必需 | 稳定且短期难以减少 | 是 |
| `flexible` | 弹性必需 | 必须发生但金额可以优化 | 是 |
| `discretionary` | 可选消费 | 可以取消、推迟或降低 | 是 |
| `exceptional` | 专项突发 | 不应作为普通月份的常态 | 否 |

基础生活支出等于 `fixed + flexible`，保守可结余空间只等于 `discretionary`。正式专项中的记录始终由专项单独归集，不进入普通月份基线；记录仍保留自己的 `spendingType`，用于专项内部复盘。

快捷记录从当前账目实时计算，场景键由 `spendingType + beneficiaryId + projectId + note` 组成，不持久化收藏、模板或最近选择。最近三个不同场景优先，其余按使用次数和最近时间排序。

## 规划数据

`cassie_plans_v5` 包含：

```json
{
  "schemaVersion": 5,
  "updatedAt": "",
  "budgets": {},
  "projects": [],
  "currentProjectId": "",
  "goals": [],
  "reviews": {},
  "noSpendDates": []
}
```

### 月度预算

以 `YYYY-MM` 为键，每月只保存：

- `availableCents`：本月可支配金额，选填。
- `totalCents`：本月日常预算，选填。
- `updatedAt`。

不再保存分类预算，也不再用历史收入推导可支配金额。
日常预算执行只累计 `fixed`、`flexible`、`discretionary` 且未关联正式专项的记录；`exceptional` 和所有正式专项支出仍进入本月总支出，但不占用日常预算。

### 正式专项

保留 `id`、`name`、`type`、`budgetCents`、`startDate`、`endDate`、`people`、`status`、`createdAt` 和 `updatedAt`。

- 类型限定为旅行、装修、节日、医疗、教育、搬家和其他。
- 状态限定为进行中或已完成。
- 只有旅行专项保存 1～20 的参与人数。
- `currentProjectId` 只能引用进行中的专项。
- 旧 `tag` 不进入 v5。

### 财务目标

保留目标 ID、名称、类型、目标金额、目标日期、状态、投入记录及时间字段。投入记录包含 ID、日期、整数分金额、20 字备注和创建时间。

### 月结

每月只保存：

- `highlight`：最多 120 字的一个结论。
- `action`：一个行动，包含 ID、最多 40 字文本、完成状态和时间字段。
- `updatedAt`。

不再保存旧 `adjustment` 或额外行动。

### 无支出日期

保存去重、升序排列的有效日期。某日新增支出后自动移除该日期。

## 完整备份 v5

```json
{
  "appName": "CassieProject",
  "backupVersion": 5,
  "schemaVersion": 5,
  "exportedAt": "2026-07-19T10:00:00.000Z",
  "summary": {
    "recordCount": 0,
    "expenseCents": 0
  },
  "records": {},
  "settings": {},
  "plans": {}
}
```

正式应用只接受 `backupVersion: 5` 和 `schemaVersion: 5`。导入流程为：选择文件、整体校验、展示摘要、明确确认、保护当前数据、写入三个主存储、失败时回滚、重新读取并核对摘要。

不再提供旧版合并账目或只覆盖账目。未知版本提示需要先转换。

## v4 到 v5 一次性转换

转换工具位于 `tools/convert-v4-to-v5.mjs`，不被正式页面加载，真实备份不进入 Git。转换规则：

- 每条 v4 `categoryId` 必须明确映射到四种 `spendingType` 之一。
- 内置默认分类映射；自定义分类通过 `--mapping` JSON 文件补充。
- 删除记录中的 `categoryId`，保留金额、日期、获益方、专项、备注和时间字段。
- 设置中移除分类定义，完整保留家庭成员、顺序、状态和默认获益方。
- 规划数据原样保留，只把结构版本更新为 5。

转换报告必须给出转换前后记录数、转换前后总支出、四种属性的笔数和未映射分类。只有未映射项为 0，且转换前后记录数与总支出都一致时才生成可导入文件。工具不会覆盖已经存在的输出文件。v3 或更早数据必须先转为 v4，再执行本次转换。

## 预测口径

- 只使用当前月之前最近三个完整自然月。
- 三个月都必须存在至少一笔普通支出，否则不输出预测。
- 每个月先排除正式专项记录和 `exceptional`，再分别汇总 `fixed`、`flexible`、`discretionary`。
- 每种属性取三个完整月的中位数；常态支出为三种中位数之和，基础生活支出为固定与弹性中位数之和。
- 下个月开始的进行中专项预算单独显示，不混入常态支出。

## 手机体验约束

- 快速记账保持“打开—输入金额—点击支出属性—完成”，默认成员或当前专项不增加必选步骤。
- 1～4 个获益方显示一行，5～8 个显示两行，不使用横向滑动。
- 高频触控目标至少 44px，表单控件字号不低于 16px。
- 支出属性使用两列四项纯文字按钮，并同时展示决策含义。
- 记账弹窗初始焦点不进入金额输入框，避免手机浏览器自动放大。
- 390×844 视口不得产生横向溢出。
- Service Worker 更新不得在用户记账过程中强制刷新，也不得混用新旧资源。
